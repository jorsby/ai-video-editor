# Grid Dimension Constraints Analysis

## 1. Why Do These Constraints Exist?

### Historical Origin: ComfyUI Grid Splitter

The constraints predate the Sharp migration (commit `70cf42a`). The original pipeline used a **fal.ai ComfyUI workflow** (`octupost/generategridimage`) to both generate AND split grid images. ComfyUI's grid splitter node likely required predictable, near-square grids to function correctly — it had no auto-detection capability and needed exact row/col parameters to divide the image into equal cells.

### The `rows === cols || rows === cols + 1` Constraint

This forces grids to be **square or nearly square** (e.g., 3x3, 4x3, 5x5, 6x5). The purpose was twofold:

1. **AI image generation quality**: Image generators (DALL-E, Flux via fal.ai) produce square or near-square images. A highly rectangular grid (e.g., 1x8) would result in very thin cells where quality degrades significantly.
2. **ComfyUI splitter compatibility**: The old splitter assumed roughly equal cell dimensions. Non-square grids could cause misalignment.

### Valid Grid Sizes in Prompts

All plan schemas list the same valid grid sizes in their system prompts:
```
2x2(4), 3x2(6), 3x3(9), 4x3(12), 4x4(16), 5x4(20), 5x5(25), 6x5(30), 6x6(36)
```

These are exactly the grids satisfying `rows === cols || rows === cols + 1` with `min=2, max=6`. The constraint is baked into the LLM instructions.

### The 2-6 vs 2-8 Split

- **Ref-to-video schemas** (wan26-flash, kling-o3, skyreels): `min(2).max(6)` — for objects and backgrounds grids. Max 36 items per grid (6x6).
- **I2V mode** (approve-grid, approve routes): `min(2).max(8)` — for scene grids. Max 64 cells (8x8), though the system prompt only lists up to 6x6.
- The i2v schema (`i2vContentSchema`) has **no Zod min/max** on rows/cols — the constraint is only enforced in the route handler.

## 2. What Are the Real Limits?

### Sharp Splitting: No Dimension Constraints

The new Sharp-based `splitGrid()` in `editor/src/lib/grid-splitter.ts` can handle **any NxM grid**:
- Uses histogram projection auto-detection to find separator lines
- Falls back to uniform division if detection fails
- Works with `rows >= 1, cols >= 1` — no inherent limit
- The only soft constraint in the detection fallback is `detectedRows >= 2 && detectedRows <= 8` (line 186-188), which is an arbitrary sanity check that should be relaxed.

### AI Image Generation Quality

The grid images are generated by fal.ai's `octupost/generategridimage` workflow. Quality depends on:

| Grid Size | Total Cells | Cell Quality | Practical Assessment |
|-----------|-------------|--------------|---------------------|
| 2x2 | 4 | Excellent | Best quality per cell |
| 3x3 | 9 | Very good | Sweet spot |
| 4x4 | 16 | Good | Still reliable |
| 5x5 | 25 | Fair | Quality starts to degrade |
| 6x6 | 36 | Marginal | Small cells, details lost |
| 7x7+ | 49+ | Poor | Too many tiny cells |

**Key insight**: Quality degrades with TOTAL cell count, not row/col ratio. A 2x5 grid (10 cells) produces better per-cell quality than a 4x4 (16 cells).

### Non-Square Grids

- **2x3, 3x2**: Work well — moderate aspect ratio, cells remain large enough.
- **1xN**: Problematic for image generation. Most image generators output square/near-square images. A 1x5 grid in a square image means 5 very wide, short cells. Quality suffers.
- **Nx1**: Same problem rotated — 5 very tall, narrow cells.
- **2xN or Nx2** (where N <= 5): Viable. The image remains reasonably proportioned.

### LLM Grid Prompt Reliability

The LLM generates prompts like `"With 2 A 3x2 Grids. Grid_1x1: ..., Grid_1x2: ..."`. LLMs can reliably:
- Generate any NxM grid format when N,M <= 6
- Generate non-square grids like 2x3, 4x2, etc.
- Struggle with very large grids (7x7+) where they lose track of cell descriptions

## 3. Recommended New Constraints

### Remove the Square Constraint

**`rows === cols || rows === cols + 1` should be REMOVED entirely.**

Rationale:
- This blocks valid useful grids like 2x3 (which is rejected because `rows(2) !== cols(3)` and `rows(2) !== cols(3)+1`)
- Sharp handles any grid dimensions
- The constraint was a ComfyUI limitation, not a quality requirement
- A story with 5 objects should be able to use a 2x3 grid (6 cells, 1 unused) OR a 3x2 grid

### New Valid Grid Sizes

Replace the rigid list with a flexible constraint:

```
MIN cells: 2 (1x2 or 2x1)
MAX cells: 36
MIN dimension: 1
MAX dimension: 8
```

**New valid grid list for system prompts:**
```
1x2(2), 1x3(3), 2x2(4), 1x5(5), 2x3(6), 1x7(7), 2x4(8), 3x3(9), 2x5(10),
3x4(12), 4x4(16), 4x5(20), 5x5(25), 5x6(30), 6x6(36)
```

However, **for practical quality**, recommend keeping the constraint simpler:

```
rows: 1-6, cols: 1-6, total cells: 2-36
```

With a warning in the UI when cells < 4 or > 25.

### Allow min=1

Single-row/column grids (1xN, Nx1) should be allowed for small item counts:
- 1x2, 1x3: Useful when user has 2-3 objects
- 2x1, 3x1: Same, just oriented differently
- However, **1x1 should remain invalid** (a "grid" of one cell is not a grid)

### Recommended Zod Schema Changes

**Ref-to-video schemas** (objects/backgrounds grids):
```typescript
// FROM: z.number().int().min(2).max(6)
// TO:   z.number().int().min(1).max(6)
// PLUS: add .refine() or route-level check: rows * cols >= 2
```

**I2V schema** (scene grids):
```typescript
// FROM: no constraint in schema, min(2).max(8) in routes
// TO:   z.number().int().min(1).max(8)
// PLUS: rows * cols >= 2 && rows * cols <= 64
```

## 4. Impact on Auto-Detection

### Sharp Auto-Detection Works Better with Hints

The current auto-detection in `detectGridLayout()` uses `expectedRows` and `expectedCols` as hints. With relaxed constraints:

- **More flexibility is fine**: Auto-detection already handles any grid. The `expectedCount` parameter in `findSeparators()` helps it choose between dark/bright separator detection.
- **Update the sanity check**: Line 186-188 in `grid-splitter.ts` limits detection to 2-8. This should be relaxed to 1-8 to support single-row/column grids:
  ```typescript
  // FROM: detectedRows >= 2 && detectedCols >= 2
  // TO:   detectedRows >= 1 && detectedCols >= 1
  ```
- **Constraints as hints, not gates**: The AI decides the grid layout, generates the image, and Sharp splits whatever it receives. The constraints should prevent clearly bad choices (too many cells, too few) but not micromanage the aspect ratio.

## 7. Risk Assessment

### Low Risk
- **Removing the square constraint**: This is purely a validation change. Sharp handles any dimensions. The LLM prompts will guide toward reasonable grids.
- **Allowing min=1**: Single-row grids work fine for splitting. The image generation quality may be slightly lower for very wide/tall images, but this is an edge case.

### Medium Risk
- **LLM generating bad grids**: Without the square constraint, an LLM might generate a 1x10 grid (10 very thin cells). Mitigation: Keep the system prompt guidance toward square grids and add a total-cells cap.
- **Image generation quality for non-square**: A 2x5 grid generates cells with ~2:1 aspect ratio. This is fine for most AI image generators but worth monitoring.

### Low Risk (but needs testing)
- **Auto-detection with 1-row grids**: The histogram projection may not find horizontal separators if there's only 1 row. But the fallback to uniform division will handle this correctly since `expectedRows=1` means no separators needed.

### Rollback Strategy
All changes are validation-level — no data model or pipeline changes. Reverting is a simple code change. Existing storyboards with valid grid sizes will continue to work unchanged.
