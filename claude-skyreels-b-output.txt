### Architecture Decision: SkyReels belongs in **ref-to-video**, not I2V

The reasoning from first principles:
- SkyReels takes **1-4 reference images** (objects + background) — this is the ref-to-video data model, not I2V's single-first-frame model
- Forcing it into I2V would require hacking the 1:1 `first_frames` relationship to store multiple images — architecturally wrong
- SkyReels maps almost 1:1 to **WAN 2.6 Flash**: flat image array, background first, simple string prompts, no `elements` API

### Key Design Decisions

1. **Own schema + system prompt** — No @Element syntax (plain-text character names), max 3 objects/scene (not 4), no multi-shot, ~80 word prompt limit
2. **Polling via decoupled cron** — `generate-video` submits to SkyReels and stores task_id. A separate `poll-skyreels` edge function (cron every 15s) checks pending tasks and writes results to DB. Client uses existing realtime subscriptions — no new client polling code
3. **Minimal DB changes** — One new column (`video_provider`) on `scenes` table. Existing `video_request_id` stores the SkyReels task_id
4. **Duration** — `bucketDuration` clamps to 1-5s. LLM targets 3-5s segments, producing more scenes (larger grids). No pipeline changes needed
5. **4 new files, 8 modified files** — detailed in the file-by-file change list
